com.apriori.collections

Class ConcurrentSortedArraySet<E>

  • java.lang.Object
    • com.apriori.collections.ConcurrentSortedArraySet<E>
  • Type Parameters:
    E - The type of element contained in the set
    All Implemented Interfaces:
    Serializable, Cloneable, Iterable<E>, Collection<E>, NavigableSet<E>, Set<E>, SortedSet<E>

    public class ConcurrentSortedArraySet<E>
extends Object
implements Serializable, Cloneable, NavigableSet<E>
    A NavigableSet that allows concurrent access. The set's elements are stored in multiple sorted arrays, each of which represents a subset (shard) of this set's elements. Multiple internal arrays are used to support concurrency with less thread contention, so an estimate for the expected level of concurrency is needed for this set to perform properly. The level of concurrency is the expected peak number of threads modifying the set simultaneously. If too small a value is used then there will be extra contention over modifying the set. If too large a value is used then there will be wasteful memory overhead.

    To allow in-process iteration to happen in the face of concurrent modifications, this set relies on the fact that the set is sorted. So calling next() on an Iterator is effectively the same as calling first() for the first element and then calling higher(lastItemFetchedViaIterator) for subsequent elements. To prevent Iterator.next() from taking O(log2n) time, iterators track version numbers for each shard and only perform O(log2n) look-ups in a shard if it has been modified since the last time it was observed by the iterator.

    Many operations that are typically constant-time operations will run in O(s) time with this set implementation, where s is the number of shards. This includes calls to methods such as first(), last(), size(), and Iterator.next().

    Similarly, query performance runs in O(s * log2(n / s)). This applies to various methods that query the set, including contains(Object), floor(Object), ceiling(Object), lower(Object), higher(Object), headSet(E), tailSet(E), and subSet(E, E). So, as the number of shards is increased from 1 to n (or even higher), query performance degrades from O(log2n) to O( greater(s,n) ).

    It is also worth pointing out that some operations that run in O(n) in a SortedArraySet, like add and remove, run in O(n / s) time in this implementation. So using a large number of shards can actually improve the performance of mutative operations (even without considering thread contention issues), but at the cost of reducing the performance of non-mutative operations.

    Generally, iteration will reflect up-to-the-moment changes to the underlying set except when a change happens between the calls to hasNext() and next(). This minor staleness is necessary to prevent next() from throwing an exception after a call to hasNext() returned true since, if the contents returned by next() were never stale, it would be possible to get such an exception if, for example, the last item in the set were removed between these two calls.

    Like many other concurrent collection implementations, operations in this set are weakly consistent. Bulk operations, like addAll(Collection) and removeAll(Collection) for example, are not guaranteed to execute atomically. So it is possible for an iteration or even a call to toString() to reflect only some of the new elements if executing concurrently with a call to addAll(Collection), for example.

    This implementation is very similar to that returned from the following expression: 

    ConcurrentSets.withNavigableSet(new SortedArraySet()).create();
    The big exception is that this implementation does not provide guarantees of strong consistency or atomicity, so as to allow greater throughput.
