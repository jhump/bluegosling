com.apriori.collections

Interface Cycle<E>

  • Type Parameters:
    E -
    All Superinterfaces:
    Collection<E>, Iterable<E>
    All Known Implementing Classes:
    LinkedCycle

    public interface Cycle<E>
extends Collection<E>
    A cyclically ordered set. An ordered sequence of objects, similar to a list, but it does not necessarily have a fixed start or fixed end. The sequence is cyclical and thus infinite. At any given time, there is a "current" element which can be thought of as the start of the sequence. To navigate the sequence, though, you can advance or retreat the current element.

    The size of the collection is the number of items added to it. This is the number of elements one can traverse before the sequence starts over.

    The standard iterator() starts iteration at the "current" element and ends before the sequence starts over. An alternate cycle() iterator is also provided. This alternate is unending (unless the sequence is empty) and repeats the sequence, following the cycle from end back to beginning.

        int advance()
        Advances the current element to the next item in the sequence. The value returned is the number of actual positions by which the current element moved.       • reset

        void reset()
        Resets the current element to the first element in the sequence. The first element starts off as the first item added to an empty sequence. When that item is the "current" item and then addFirst(Object) is invoked, the newly added item becomes the first element. If the first element is ever removed, the item after it is the first item.

        If the sequence is empty, calling this method does nothing.

      • reverse

        Cycle<E> reverse()
        Returns a view of this sequence in reverse order. The current element remains unchanged, but the sequence around it will be reversed. What would previously have been the next element will be the previous one in the returned view and vice versa. Calling reverse() on the returned cycle will return the original sequence.

        Changes made to the view, including moving the "current" cursor, will be visible through the original cycle. For example, calling advance() on the returned view has the same effect as having called retreat() on the original sequence; calling addFirst(Object) on the view has the same affect as calling addLast(Object) followed by retreat() on the original.

        Returns:
        a view of the cycle in reverse order
